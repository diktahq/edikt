# ADR-007: Compile Schema Version for Generated Governance

**Date:** 2026-04-09
**Status:** Accepted

## Context

`/edikt:gov:compile` produces `.claude/rules/governance.md` from a project's ADRs, invariants, guidelines, and rule packs. That output needs a version stamp so downstream tools can answer questions like:

1. *Is this governance stale? Should I regenerate?*
2. *Can edikt read this file's format?*
3. *What produced it, and when?*

Before v0.2.3, compile stamped `version: "{edikt_version}"` into the generated governance.md frontmatter — i.e. the edikt marketing version at compile time. A dogfood test enforced that this field equalled the project's `VERSION` file.

This conflated two unrelated cadences:

- **edikt marketing version** changes on every release, including pure bug fixes (e.g. v0.2.0 → v0.2.1 touched init.md and review.md but did not change compile output format).
- **compile output format** changes rarely — only when the structure of generated rule files materially changes (e.g. v0.1.x flat governance.md → v0.2.x topic-grouped rule files).

Treating them as the same field produced two concrete problems:

1. **False-positive staleness signals.** Every point release made every user's governance "stale" by the version check, even though nothing about compile had changed. No regeneration was actually needed.
2. **Dogfooding drift.** In the edikt repo itself, we hand-edited `governance.md`'s `version:` field via `sed` on every release to keep the dogfood test green, without actually re-running `/edikt:gov:compile`. The file's `<!-- compiled: ... -->` timestamp diverged from reality — the version said v0.2.2 but the timestamp was frozen at v0.2.0's compile date. The file was lying about its own provenance.

## Decision

Introduce a **compile schema version** — a small integer, independent of edikt's marketing version, that identifies the output format contract of `/edikt:gov:compile`.

```yaml
---
paths: "**/*"
compile_schema_version: 2
---
<!-- edikt:compiled — generated by /edikt:gov:compile, do not edit manually -->
<!-- compiled_by: edikt v0.2.3 -->
<!-- compiled_at: 2026-04-09T10:30:00Z -->
```

**Three separate fields, three separate purposes:**

| Field | Purpose | Enforced by |
|---|---|---|
| `compile_schema_version` (YAML frontmatter) | Programmatic — can edikt read this file? Should it be regenerated? | `/edikt:doctor` compares against the constant in `commands/gov/compile.md`. |
| `compiled_by` (HTML comment) | Human diagnostic — what edikt version last ran compile here? | Informational only. Not enforced. |
| `compiled_at` (HTML comment) | Human diagnostic — when? | Informational only. Not enforced. |

**Current schema versions:**

- **`compile_schema_version: 1`** — v0.1.x flat `governance.md` (single file, one directive per line, 30-directive cap).
- **`compile_schema_version: 2`** — v0.2.x topic-grouped rule files under `governance/` directory, with routing table index, directive sentinels, and no directive cap.

The schema version is a constant declared at the top of `commands/gov/compile.md` and only bumped when the output *structure* changes in a way that old tooling cannot read. Point releases that fix bugs, add prose to ADRs, or touch unrelated commands do **not** bump the schema.

**`/edikt:doctor` behavior:**

- If `compile_schema_version` is missing → governance was generated by v0.1.x. Recommend `/edikt:gov:compile` to regenerate in v2 format.
- If `compile_schema_version` < current → output format has changed. Recommend regeneration.
- If `compile_schema_version` > current → installed edikt is older than what produced this file. Recommend upgrading edikt.
- If equal → no staleness signal from schema version. (Actual staleness is a different question — handled by ADR/invariant mtime comparisons.)

## Why not an incremental counter?

A per-project incremental counter (`compile_revision: 47`) would track "how many times has compile run in this project", but it doesn't answer any question the user cares about. There's no reference value to compare against — nothing else knows what the counter *should* be. It only catches hand-editing (and poorly, since the user would just bump the counter too).

A schema version is meaningful because it's a **contract** between the compile command and its output, checkable without any external state.

## Why keep `compiled_by` and `compiled_at` as HTML comments?

These are for humans opening the file to answer "when was this last compiled and by what?" They must not be enforced programmatically — if they were, we'd recreate the original problem of tying governance staleness to edikt marketing version. Keeping them in HTML comments (invisible in markdown preview but visible in raw file) signals "diagnostic, not contract".

## Migration

- v0.2.2 and earlier installations will have `version: "<edikt-version>"` in their generated governance.md.
- `/edikt:doctor` detects this legacy field and recommends recompiling.
- `/edikt:gov:compile` treats any file without `compile_schema_version` as schema 1 (legacy) and rewrites it to the current schema on next compile.
- The test that enforced `GOV_VER == FILE_VER` is removed. A new test verifies `compile_schema_version` is a valid integer matching the constant in `commands/gov/compile.md`.

## Consequences

- **No more noisy staleness warnings** on point releases. Users are only told to recompile when the output format actually changed.
- **Honest provenance** — the dogfood repo stops hand-editing governance.md on release. Either we actually re-run `/edikt:gov:compile` (keeping timestamp + compiled_by accurate) or we leave the file alone.
- **Decoupled release cadences** — edikt can ship bug fixes without implying anything about compile output compatibility.
- **One more field to write in compile output** — but this is a small cost for clear semantics.
- **`/edikt:doctor` gets smarter** — it can now distinguish "stale for real reasons" from "stale because edikt marketing version ticked".
