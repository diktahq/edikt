# ADR-019: Security rewrites may change hook content and wrapping in the same commit — narrow carve-out of ADR-014

**Status:** Accepted
**Date:** 2026-04-17
**Decision-makers:** Daniel Gomes

---

## Context and Problem Statement

ADR-014 directs hook JSON-wrapping migrations to preserve user-visible message content byte-for-byte. Its directive: "NEVER reword user-facing strings as part of the wrapping migration. Content changes are separate commits with separate fixtures." The purpose is to keep mechanical migrations small and reviewable, and to ensure fixture regeneration captures exactly the protocol change.

The v0.5.0 security audit (2026-04-17) identified four hooks whose current user-visible content is itself the vulnerability:

- `subagent-stop.sh` — the `GATE_MSG` instructs Claude to execute shell with an attacker-influenceable finding interpolated. Keeping the content byte-for-byte through the JSON-wrapping migration would ship a Critical (CRIT-1) vulnerability. The only correct fix is to make the `systemMessage` static and move the log-write inside the hook process itself — a content change.
- `file-changed.sh` — systemMessage concatenates the attacker-controlled file path (CRIT-5). Safe emission requires a `json.dumps`-based write AND a static-phrasing of the message to avoid any downstream shell use.
- `headless-ask.sh` — permissionDecision output is concatenated from a config-supplied answer (CRIT-4). Safe emission requires structured construction, which necessarily changes how the answer is rendered.
- `stop-failure.sh` — error.type/error.message fields are concatenated into JSON; the safe pattern changes the event-log payload structure (CRIT-2).

These four hooks cannot be safely JSON-wrapped under ADR-014's strict byte-for-byte rule without shipping open Critical findings. This ADR carves out a narrow exception.

## Decision Drivers

- INV-003 and INV-004 are non-negotiable. Any hook whose content violates them must be fixed before v0.5.0 ships.
- ADR-014's rule exists for a good reason — it prevents content drift during mechanical migrations. That reasoning holds for every hook that does NOT have a current INV-003/004 violation.
- A blanket supersession of ADR-014 would be over-broad. The scope is four specific hooks with specific violations.
- Honoring ADR-014 strictly would require two PRs per affected hook (wrap first with a known-vulnerable message, then rewrite the message in a follow-up). That's more commits and more risk, not less.

## Considered Options

1. **Narrow carve-out naming exactly four hooks** — the chosen approach.
2. **Fully supersede ADR-014** — replace its byte-for-byte rule. Too broad.
3. **Land security rewrites as separate post-migration PRs per ADR-014** — doubles PR count; ships known vulnerabilities between the two.
4. **Block security rewrites until ADR-014 migration is complete** — worse; keeps Criticals open for the migration window.

## Decision

ADR-014's byte-for-byte rule is narrowed by this ADR for exactly four hooks: `templates/hooks/subagent-stop.sh`, `templates/hooks/file-changed.sh`, `templates/hooks/headless-ask.sh`, `templates/hooks/stop-failure.sh`. For these four hooks only, security-driven rewrites may change user-visible message content AND JSON wrapping format in the same commit, with fresh fixtures generated by running the updated hook against its payload.

No other hook may invoke this carve-out. Any future content change to a non-listed hook must follow ADR-014 unaltered (wrap first, content change separately).

The carve-out is justified only when the existing content violates INV-003 or INV-004. Style rewrites, prose polish, or UX improvements do not qualify.

## Alternatives Considered

### Full supersession of ADR-014
- **Pros:** One rule instead of two.
- **Cons:** Removes a protection that is still valid for the majority of hooks. Opens the door to content drift during routine migration work.
- **Rejected because:** ADR-014's rule is still correct for mechanical migrations; a narrow carve-out is more honest about what changed.

### Separate PRs per affected hook
- **Pros:** Strictly honors ADR-014.
- **Cons:** Ships a known-Critical vulnerability between the wrapping PR and the security rewrite PR. That's worse than a single well-reviewed PR that does both.
- **Rejected because:** the gap between PRs is exactly where an attacker would attack.

## Consequences

- **Good:** CRIT-1, CRIT-2, CRIT-4, CRIT-5 can be fixed in a single well-reviewed change per hook without intermediate vulnerable states.
- **Good:** ADR-014 remains in force for every other hook. The carve-out is explicit and narrow.
- **Bad:** Four hooks have simultaneous protocol + content changes. Reviewers must verify both the JSON structure and the new static message text in the same diff.
- **Bad:** Introduces a precedent — "security can override byte-for-byte rules." Mitigated by listing the four hooks exhaustively. Future applications of this carve-out require a new ADR.
- **Neutral:** ADR-014 is not superseded. Both ADRs remain accepted, and the hook migration plan in PLAN-v0.5.0-stability.md continues under ADR-014 for all other hooks.

## Confirmation

- `templates/hooks/subagent-stop.sh`, `file-changed.sh`, `headless-ask.sh`, `stop-failure.sh` have been rewritten under this carve-out with fresh characterization fixtures regenerated per ADR-011's regen rule.
- No other hook under `templates/hooks/` has a content change bundled with its JSON-wrapping migration. Verified by `git log -p templates/hooks/*.sh` on the v0.5.0 branch showing that non-listed hooks have wrap-only diffs.
- Both ADR-014 and ADR-019 remain `status: accepted`. Neither supersedes the other.
