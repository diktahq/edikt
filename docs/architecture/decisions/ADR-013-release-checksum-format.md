# ADR-013: Release artifacts use an aggregated SHA256SUMS file, not per-file sidecars

**Status:** Superseded by ADR-016
**Date:** 2026-04-16
**Decision-makers:** Daniel Gomes

---

## Context and Problem Statement

Every v0.5.0 GitHub Release will ship two tarballs: `edikt-v<ver>.tar.gz`
(launcher only — `bin/edikt`, `LICENSE`, `README.md`) and
`edikt-payload-v<ver>.tar.gz` (templates, commands, hooks). Both must be
verifiable by the Homebrew formula and by `bin/edikt install` when the user
runs it outside of Homebrew.

`bin/edikt` already implements `verify_against_sidecar()` (line 87) which
accepts either a bare hex hash string or a `<hash>  <filename>` line in the
standard `sha256sum` format. SPEC-004 §4 required a format decision before
wiring the release workflow, because the workflow emitter and the launcher's
verification logic must agree on the shape of the checksum artifact.

How should the release workflow publish checksums for the two release tarballs?

## Decision Drivers

- `verify_against_sidecar()` already handles the `sha256sum` line format —
  no new parsing code needed if we use that format
- The Homebrew formula embeds a single `sha256` value directly in the Ruby DSL;
  the formula does not read an external checksum file
- Standard practice in the Go and Rust ecosystems is a single `SHA256SUMS` file
  listing all release artifacts, one line per artifact
- Simplicity: the release workflow calls `sha256sum *.tar.gz > SHA256SUMS` —
  one command, one artifact to upload, no per-file naming convention needed
- Future signing (GPG, Sigstore) is explicitly out of scope for v0.5.0 and can
  be added as a separate artifact without changing the `SHA256SUMS` format

## Considered Options

1. **Aggregated `SHA256SUMS`** — one file, all artifacts, `sha256sum` format
2. **Per-file sidecars** — `edikt-v<ver>.tar.gz.sha256` and `edikt-payload-v<ver>.tar.gz.sha256`
3. **Inline in GitHub Release body** — no separate file; checksums in the release description

## Decision

We will ship a single `SHA256SUMS` file as a GitHub Release asset. The file
contains one line per tarball in the standard `sha256sum` format:

```
<64-hex-chars>  edikt-v0.5.0.tar.gz
<64-hex-chars>  edikt-payload-v0.5.0.tar.gz
```

Generated by: `sha256sum edikt-v${VERSION}.tar.gz edikt-payload-v${VERSION}.tar.gz > SHA256SUMS`

`bin/edikt install` MUST fetch `SHA256SUMS` alongside the payload tarball and
verify the launcher tarball's hash before extracting. The verification uses the
existing `verify_against_sidecar()` function, which greps the `SHA256SUMS`
content for the matching filename.

Signing (GPG / Sigstore / cosign) is deferred to a future ADR. The `SHA256SUMS`
format is compatible with any future signing approach — Sigstore's `cosign`
can sign the `SHA256SUMS` file itself, and GPG detached signatures work the
same way.

## Alternatives Considered

### Per-file sidecars
- **Pros:** Each tarball carries its own checksum; simpler for tools that expect `<artifact>.sha256`
- **Cons:** Two separate files to manage; no standard naming convention; `verify_against_sidecar()` would need to know the sidecar filename; release asset count grows linearly with artifact count
- **Rejected because:** More complexity, no benefit over aggregated `SHA256SUMS`

### Inline in GitHub Release body
- **Pros:** No extra files; visible in the GitHub UI
- **Cons:** Not machine-readable; breaks automated verification; not supported by Homebrew formula verification
- **Rejected because:** Not suitable for automated pipeline consumption

## Consequences

- **Good:** Single `SHA256SUMS` file; release workflow is one `sha256sum` command; compatible with future signing
- **Good:** `verify_against_sidecar()` works unchanged — it already parses `<hash>  <filename>` lines
- **Bad:** `bin/edikt install` must download `SHA256SUMS` in addition to the payload tarball (one extra HTTP request)
- **Neutral:** Homebrew formula embeds the launcher SHA256 directly in the `.rb` file — the formula does not read `SHA256SUMS`

## Confirmation

- `test/integration/release/test_tarball_contents.sh` verifies that `SHA256SUMS` contains both tarball filenames and that each hash is 64 hex characters
- `bin/edikt install` fetches and verifies against `SHA256SUMS` (existing `verify_against_sidecar` path)
- `.github/workflows/release.yml` uploads `SHA256SUMS` as a release asset alongside the two tarballs
