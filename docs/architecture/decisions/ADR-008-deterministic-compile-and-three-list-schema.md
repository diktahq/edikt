# ADR-008: Deterministic Compile and Three-List Directive Schema

**Date:** 2026-04-09
**Status:** Accepted
**Supersedes:** None
**Superseded by:** None

## Context

`/edikt:adr:compile`, `/edikt:invariant:compile`, and the new `/edikt:guideline:compile` (added in v0.3.0 for parity) generate LLM-readable enforcement directives from the bodies of compiled artifacts. Before v0.3.0, the directives block had two critical problems:

1. **Stochastic output.** Every compile run produced slightly different directive wording for the same body. Suppression by text match was impossible. Hand-edit detection fired false positives. Git churn appeared on every compile even when nothing conceptual changed.

2. **No mechanism for user additions, corrections, or removals.** If compile missed a rule, the user could hand-edit it into `directives:`, but the next compile would overwrite the edit. If compile generated a wrong rule (hallucinated, mis-extracted, or nuance-shifted), the user had no way to permanently remove it — it would come back on every regeneration. Users either trusted compile blindly or stopped using it.

This ADR locks in a formal contract that solves both problems simultaneously, across all three artifact types (ADR, invariant, guideline).

## Decision

### 1. Three-list directive schema

Every compiled artifact's `[edikt:directives:start]: #` / `[edikt:directives:end]: #` block contains three declarative lists plus metadata:

```yaml
[edikt:directives:start]: #
source_hash: <sha256 hex>         # required after first compile
directives_hash: <sha256 hex>     # required after first compile
compiler_version: "<semver>"      # required after first compile
paths:
  - <glob>
scope:
  - <scope tag>
directives:
  - "<rule text>"
manual_directives:
  - "<rule text>"
suppressed_directives:
  - "<rule text>"
[edikt:directives:end]: #
```

**List semantics:**

- **`directives:`** — Auto-generated by `<artifact>:compile` from the artifact body. Rewritten on every body change. Hand-edits are detected and resolved via the interview flow. This list is the output of Claude's reading of the body.
- **`manual_directives:`** — User-authored rules. NEVER touched by `<artifact>:compile`. Preserved across all regenerations. Used when compile missed something the user wants enforced.
- **`suppressed_directives:`** — Auto directives the user rejects. NEVER touched by `<artifact>:compile`. Used when compile keeps generating something wrong, hallucinated, or nuance-shifted. Filtering happens at `gov:compile` time, not at `<artifact>:compile` time.

**All three lists are optional.** A block without `manual_directives:` or `suppressed_directives:` keys is valid — both default to empty lists. This preserves backward compatibility with v0.2.x artifacts.

### 2. Responsibility split between compile commands

Two commands process directive blocks, with strictly separated roles:

| Command | Reads | Writes | Calls Claude? |
|---|---|---|---|
| `<artifact>:compile` | artifact body, stored hashes, existing `directives:` list | `directives:` list, `source_hash`, `directives_hash`, `compiler_version` | Yes — to regenerate auto directives from body |
| `gov:compile` | all three lists + `paths`/`scope` from every artifact | `.claude/rules/governance.md` and topic files | No — pure merge/filter |

**`<artifact>:compile` NEVER:**
- Reads `manual_directives:`
- Reads `suppressed_directives:`
- Modifies either of those lists
- Filters `directives:` based on `suppressed_directives:`
- Merges `directives:` with `manual_directives:`

**`gov:compile` NEVER:**
- Writes to artifact files
- Modifies any of the three lists in place
- Triggers re-generation of auto directives

**`gov:compile` processing per artifact:**

```
effective_rules = (directives - suppressed_directives) ∪ manual_directives
```

Set semantics: `suppressed_directives` filters by exact string match against `directives`; `manual_directives` unions in after filtering. De-duplication happens at the merge step across all artifacts (if the same rule appears in multiple artifacts, it lands once in governance.md).

This split ensures:
- User edits to `manual_directives:` or `suppressed_directives:` flow through to governance.md on the next `gov:compile` run without requiring `<artifact>:compile` to run.
- `<artifact>:compile` can use hash-based caching without missing user intent.
- Artifact files are declarative; effective state is computed at application time.

### 3. Hash-based caching

Two hashes are stored in the directives block to enable deterministic caching:

**`source_hash`** — SHA-256 of the artifact body with the directives block excluded.

Normalization, in order:
1. Read the full artifact file as UTF-8 text.
2. Remove the entire `[edikt:directives:start]: #` … `[edikt:directives:end]: #` block, inclusive of both sentinel lines.
3. Replace all `\r\n` and `\r` with `\n` (normalize line endings).
4. For each line, strip trailing whitespace (`[ \t]+$`).
5. Compute SHA-256 of the resulting bytes. Encode as lowercase hexadecimal.

**`directives_hash`** — SHA-256 of the canonicalized `directives:` list (auto directives only, not `manual_directives:` or `suppressed_directives:`).

Canonicalization:
1. Parse the YAML block and extract the `directives:` list.
2. Join the list items with a single `\n` character (LF), in document order.
3. Compute SHA-256 of the resulting bytes. Encode as lowercase hexadecimal.

**`compiler_version`** — String matching the `edikt_version` at the time of compile (e.g., `"0.3.0"`). Used to detect algorithm drift across edikt versions. Default behavior on version mismatch: warn, do not auto-regenerate.

### 4. Compile algorithm (`<artifact>:compile`)

```
read artifact file
current_source_hash       = compute source_hash from current body
stored_source_hash        = extract from block (may be absent for legacy)
current_directives_hash   = compute directives_hash from current directives: list
stored_directives_hash    = extract from block (may be absent for legacy)

if stored_source_hash is absent OR stored_directives_hash is absent:
    → first compile: run Claude to generate directives from body,
      write new directives: list, compute and write both hashes,
      write compiler_version. Done.

if current_source_hash == stored_source_hash
   AND current_directives_hash == stored_directives_hash:
    → FAST PATH: output "ADR-NNN: up to date". No Claude call. No writes. Done.

if current_source_hash != stored_source_hash:
    → body changed: run Claude to regenerate directives from new body.
      Write new directives: list. Compute and write both new hashes.
      Update compiler_version. If regeneration produces output that differs
      from what the user hand-edited before, trigger interview.

if current_source_hash == stored_source_hash
   AND current_directives_hash != stored_directives_hash:
    → body stable but user hand-edited directives.
      Trigger interview (see Section 5).
      After resolution, compute new directives_hash and update.
      source_hash is unchanged.
```

**`manual_directives:` and `suppressed_directives:` never factor into the hash comparison** — they are not hashed, never trigger `<artifact>:compile` to run, and never alter the cache decision.

**`--regenerate` flag** forces a full Claude re-run regardless of hash match. Used when the user explicitly wants fresh output (e.g., after an edikt version upgrade that changed compile logic).

### 5. Hand-edit interview

When `<artifact>:compile` detects that `directives_hash` does not match the current `directives:` list (case c in Section 4), it enters an interactive interview, one question per detected change.

**For each hand-added line in `directives:`:**

```
Found hand-added line in directives:
    "<line text>"

This line is in directives: but wouldn't be generated from the current body.
What should I do?

  [1] Move to manual_directives: (keep as user-authored rule)
  [2] Add to suppressed_directives: (pick this only if compile had generated
      this before and you want to prevent regeneration)
  [3] Delete entirely
  [4] Edit artifact body first so this becomes an auto directive
  [5] Skip for now (leave in directives: — will be overwritten next compile)

Choice [1]:
```

**For each deleted auto directive (line that compile wants to regenerate but was removed):**

```
Found deleted auto directive:
    "<line text>"

Compile still wants to regenerate this from the body. What should I do?

  [1] Add to suppressed_directives: (keep deleted — compile filters it out)
  [2] Let compile regenerate (deletion was accidental)
  [3] Edit artifact body first to remove the source of this directive

Choice [1]:
```

**No silent auto-resolution.** Every detected edit produces an explicit user choice. Claude waits for each response before moving on.

**Headless mode behavior:**

When the command runs without an interactive terminal (no `/dev/tty` or `EDIKT_HEADLESS=1`), the interview cannot proceed. Instead:

```
❌ /edikt:<artifact>:compile <ID>

Cannot run in headless mode: hand-edited lines detected requiring
user intervention.

  + "<added line>"
  - "<deleted line>"

Options:
  1. Run interactively: /edikt:<artifact>:compile <ID>
  2. Override (destructive):  --strategy=regenerate
       → Discards hand-edits, rewrites directives: from body
  3. Override (preserve):     --strategy=preserve
       → Leaves file unchanged, skips this artifact

Exiting with error (2).
```

**`--strategy=regenerate`** — discard hand-edits, run Claude, write fresh directives. `manual_directives:` and `suppressed_directives:` are still preserved.

**`--strategy=preserve`** — do nothing, exit 0. Used when CI wants to skip ambiguous artifacts and report them elsewhere.

### 6. Auto-chain from `<artifact>:new`

`/edikt:<artifact>:new` automatically runs `/edikt:<artifact>:compile <ID>` at the end of its workflow. On a freshly created artifact, there's nothing to preserve, no manual additions, no hand-edits — the interview never fires. The user never has to remember to run compile after new.

## Migration from v0.2.x

v0.2.x artifacts have directive blocks without the new `manual_directives:`, `suppressed_directives:`, `source_hash`, `directives_hash`, or `compiler_version` fields. These are all optional and default to empty lists or absent values.

**First compile after v0.3.0 upgrade:**

1. Read artifact. Note the absence of hashes.
2. Run Claude to regenerate `directives:` from body.
3. Write new `directives:` list.
4. Compute and write `source_hash`, `directives_hash`, `compiler_version`.
5. `manual_directives:` and `suppressed_directives:` are written as empty lists.

No user intervention required for migration. Existing v0.2.x artifacts continue to work unchanged until the user next runs `<artifact>:compile` on them.

## Consequences

**Positive:**

- **Deterministic output.** Once committed, directives never drift on re-compile. Git churn eliminated.
- **User edits are preserved.** `manual_directives:` and `suppressed_directives:` flow through safely forever.
- **User edits are reversible.** Suppression can be undone by removing the entry from `suppressed_directives:`.
- **Fast re-compile.** Hash-based skip means pre-commit/pre-push hooks can run `<artifact>:compile` on every artifact with near-zero cost.
- **Clean separation of concerns.** `<artifact>:compile` handles generation; `gov:compile` handles application. Neither has implicit knowledge of the other.
- **Backward compatible.** v0.2.x artifacts migrate silently on first compile.

**Negative:**

- **Schema is larger.** Three new lists plus three metadata fields. The block is more verbose.
- **First compile is still stochastic.** Two users running compile on a fresh artifact in parallel may produce different initial wording. Whoever commits first wins; the committed version becomes canonical.
- **Hand-editing `directives:` requires an interview.** Adds friction for power users who know what they want. Mitigated by `--strategy=regenerate` for full rewrites.
- **Another two YAML lists in every block.** Users must learn which list to use for which intent. Mitigated by the decision-tree comment in template files.
