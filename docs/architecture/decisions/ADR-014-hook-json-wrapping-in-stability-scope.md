# ADR-014: Hook JSON-wrapping migration is in-scope for v0.5.0 stability release

**Status:** accepted
**Date:** 2026-04-16
**Decision-makers:** Daniel Gomes

---

## Context and Problem Statement

ADR-011 (2026-04-15) deferred hook protocol migration to v0.6.0 on the basis that
"rewriting 9 hooks is a significant behavioral change scope" incompatible with v0.5.0's
stability mandate. One day later, a parity review against Claude Code v2.1.72–v2.1.111
surfaced that only 7 of edikt's 16 hooks actually emit plaintext to stdout today
(`pre-compact`, `pre-tool-use`, `session-start`, `post-tool-use`, `post-compact`,
`subagent-stop`, `stop-failure`, `user-prompt-submit`); the rest either already emit JSON
(`phase-end-detector`, `headless-ask`, `file-changed`, `stop-hook`) or are log-only side
effects with no stdout (`cwd-changed`, `instructions-loaded`, `event-log`, `task-created`).

The migration is not a "rewrite" — it is a mechanical wrapping: `echo 'x'` becomes
`printf '{"systemMessage":"%s"}\n' 'x'`. Claude Code's hook protocol renders
`systemMessage` to the same user-visible surface as unstructured stdout. The wrapping
preserves observable UX while gaining protocol compliance.

Separately, v0.6.0 is being re-scoped to the full SDLC rework. Keeping hook protocol
work in v0.6.0 muddies that theme and splits parity work across two releases when it
could land as a single coherent v0.5.0 parity deliverable.

How should edikt sequence hook JSON protocol migration relative to the v0.5.0 stability
release?

## Decision Drivers

- v0.6.0 thematic coherence — SDLC rework deserves a clean release window
- Stability mandate of v0.5.0 — hook work must not introduce observable UX regressions
- Characterization test net already in place — regressions would be caught before ship
- Parity coherence — mixing "some hooks migrated, others not" across releases is worse
  than a single bundled migration

## Considered Options

1. **Keep ADR-011 as-is** — hook migration ships in v0.6.0, parity work splits across releases
2. **Supersede ADR-011** — reframe JSON-wrapping as compatibility-preserving, include in v0.5.0
3. **Partial migration** — migrate only the simplest hooks in v0.5.0, defer complex ones

## Decision

Hook JSON-wrapping migration IS in-scope for v0.5.0, treated as a compatibility-preserving
wrapper change rather than a behavioral rewrite. Specific consequences locked in by this
decision:

- All plaintext-emitting edikt hooks MUST emit JSON conforming to the Claude Code hook
  protocol (`{"systemMessage": ...}`, `{"additionalContext": ...}`, `{"continue": ...}`,
  `{"decision": ...}`) by the v0.5.0 release. The user-visible message text MUST be
  preserved byte-for-byte inside the wrapping — migration is a transport change, not a
  content change.
- The characterization test gate (`EDIKT_SKIP_HOOK_TESTS=1` opt-out) from ADR-011 MUST
  remain opt-out. ADR-011's test infrastructure decisions (opt-out polarity, `_runner.sh`
  immutability, `_staged_runner.sh` extension, fixture `_note` rationale) carry forward
  unchanged.
- The 3 fixture pairs removed by ADR-011 are resolved as follows:
  - `pre-compact` — NOT re-added (hook permanently deleted in §14.1).
  - `session-start-with-edikt` — re-addition pending `_staged_runner.sh` extension to
    provision git history + deterministic `MEMORY.md`. Tracked as v0.5.x follow-up;
    not a v0.5.0 ship-gate.
  - `subagent-stop-critical` — re-addition pending stable test env vars for git
    identity and clock. Tracked as v0.5.x follow-up; not a v0.5.0 ship-gate.

  When re-added, each fixture MUST document its `verified_by` command. The v0.5.0
  ship-gate is the hook-level JSON migration (§14.1) plus fixtures.yaml §9.1 notes
  documenting the deferral rationale for the 2 pending pairs.
- Fixture expected outputs for migrated hooks MUST be regenerated by running the updated
  hook against its payload (`cat payload.json | bash hook.sh`). Hand-authored expected
  outputs remain forbidden per ADR-011.
- Hook behavior refinements unblocked by lifting the v0.6.0 gate — `pre-tool-use.sh`
  `updatedInput` transformation, `task-created.sh` plan-phase tracking, and new hook
  events (`SessionEnd`, `SubagentStart`, `TaskCompleted`, `WorktreeCreate`,
  `WorktreeRemove`) — are permissible in v0.5.0 provided each is covered by a
  characterization fixture pair before ship.
- Migration MUST NOT change message content. If a user-facing string needs rewording,
  that is a separate commit with a separate fixture update, not folded into the wrapping
  migration.

## Alternatives Considered

### Keep ADR-011 as-is

- **Pros:** ADR-011's stability reasoning is defensible on its own terms; no governance
  churn; v0.6.0 already carries the hook test infrastructure work.
- **Cons:** Parity work splits across v0.5.0 (tracker file, `initialPrompt`, statusline,
  docs) and v0.6.0 (all hook behavior). v0.6.0's SDLC rework theme gets diluted. Users
  on v0.5.0 get a partial parity story.
- **Rejected because:** the "stability risk" argument in ADR-011 overestimates the
  migration surface once the plaintext-vs-JSON audit is done. Wrapping preserves UX;
  characterization tests catch regressions; cost is ~2–3 days focused work.

### Partial migration

- **Pros:** Lower risk envelope — migrate only `pre-compact` (being deleted anyway) and
  `pre-tool-use` (7 lines) in v0.5.0, defer larger hooks.
- **Cons:** Leaves edikt in a mixed state (some hooks JSON, some plaintext) for a release
  cycle — worst of both worlds for test suite, documentation, and user expectations.
- **Rejected because:** coherence matters more than incrementalism here. Either the
  argument "wrapping is non-breaking" holds (migrate all) or it doesn't (migrate none).

## Consequences

- **Good:** All parity work lands in v0.5.0 as a single coherent release. Hook protocol
  work unblocks the 3 previously-excluded fixtures, restoring full test coverage.
- **Good:** v0.6.0 is freed to focus exclusively on SDLC rework without inherited hook
  scope.
- **Good:** Future hook behavior additions (new events, `updatedInput` transforms) become
  permissible under normal change control rather than gated on a release boundary.
- **Bad:** v0.5.0 scope grows by ~2–3 days of focused work plus test regeneration time.
  SPEC-004 must be updated to reflect the expanded scope.
- **Bad:** Introduces a precedent — "compatibility-preserving migrations are in-scope for
  stability releases." Future ADRs must hold the line on what qualifies as
  "compatibility-preserving" to prevent scope creep on stability releases generally.
- **Neutral:** ADR-011 supersession creates a governance chain users must follow. Both
  ADRs remain in the decision log; ADR-011's reasoning is preserved as historical
  context.

## Confirmation

- All stdout-producing hooks produce valid JSON on their primary output path. Note:
  `post-tool-use.sh` and `stop-failure.sh` are silent-exit hooks (no stdout — they
  trigger external formatters / write to `events.jsonl` as side effects). JSON
  wrapping applies only to hooks that emit to stdout.
  Verified by:
  ```bash
  for h in pre-tool-use session-start post-compact subagent-stop user-prompt-submit; do
    cat test/fixtures/hook-payloads/${h}*.json 2>/dev/null | bash templates/hooks/${h}.sh \
      | python3 -c 'import json,sys; json.loads(sys.stdin.read())' || echo "FAIL: $h"
  done
  ```
  `pre-compact.sh` is NOT in this list — it was deleted per §14.1. Attempting to
  run the deleted hook would fail; its absence is verified by
  `test ! -f templates/hooks/pre-compact.sh`.
- `fixtures.yaml` §9.1 must document the resolution of the 3 removed pairs:
  `pre-compact` permanently removed (hook deleted); `session-start-with-edikt` and
  `subagent-stop-critical` deferred with explicit `_note` explaining the blocker.
  ```bash
  grep -E '(pre-compact|session-start-with-edikt|subagent-stop-critical)' \
    docs/product/specs/SPEC-004-v050-stability/fixtures.yaml
  ```
  Each deferred pair MUST cite the blocker (e.g., `_staged_runner.sh` extension
  requirement). Full re-addition tracked as v0.5.x follow-up.
- Hook test suite MUST pass without `EDIKT_SKIP_HOOK_TESTS=1`:
  ```bash
  for t in test/unit/hooks/test_*.sh; do bash "$t"; done
  ```
- No user-facing message content changes between pre- and post-migration fixtures. Verified
  by a `git log -p test/expected/hook-outputs/` review showing only JSON wrapping diffs,
  not string-content diffs.
- ADR-011's Status line MUST read "Superseded by ADR-014" in both frontmatter
  (`status: superseded-by-ADR-014`) and the bolded Status line.
