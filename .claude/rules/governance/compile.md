## Prohibitions
- MUST NOT attach prohibitions as a child `prohibits[]` array on a parent directive — prohibitions are synthesised from rejected `## Considered Options`, not from chosen directives, so there is no parent directive to attach them to. (ref: ADR-032)
- MUST NOT cache subagent outputs to disk for --check determinism — sidecars are the cache; adding a cache on top of a cache introduces stale-cache, cross-machine, and corruption failure modes. (ref: ADR-028)
- MUST NOT fold prohibitions into `directives[]` with a `kind` discriminator — `len()` no longer answers 'how many prohibitions' and the INV-005 byte-range guard cannot cleanly carve three managed regions without ad-hoc per-entry markers. (ref: ADR-032)
- MUST NOT implement compile as a cache layer only — first compile per artifact still pays full latency cost and a cache layer does not fix determinism across LLM runs. (ref: ADR-020)
- MUST NOT implement resync as a separate user-driven verb (edikt resync) — creates a two-step UX for the common case and new users will experience compile as broken when sidecars are stale. (ref: ADR-028)
- MUST NOT place prohibitions in a free-form annotation field — doing so defeats `additionalProperties: false` strictness, `KnownFields(true)` decoder safety, and every mechanical check. (ref: ADR-032)
- MUST NOT rely on better prompting of the LLM compile path as a substitute for tier-2 migration — it cuts latency ~2x at best and does not solve determinism or token cost. (ref: ADR-020)
- MUST NOT remove LLM involvement from compile entirely — users who write a new ADR must not be required to author the sentinel block by hand; auto-generation is a headline product feature. (ref: ADR-020)
- MUST NOT resync inline while skipping determinism for --check — CI cannot rely on --check being deterministic and two CI runs of the same input would produce different verdicts. (ref: ADR-028)
- MUST NOT store sidecars in a separate directory (e.g., `.edikt/sidecars/<name>.yaml`) — doing so loses co-location at the directory level and breaks `git mv` ergonomics. (ref: ADR-027)
- MUST NOT use a one-phase compile that always re-runs subagents inline — this blows ADR-020's budget on every steady-state compile and makes --check either nondeterministic or self-contradictory. (ref: ADR-028)
- MUST NOT use a single project-level sidecar for all governance metadata — doing so loses co-location, git-blame locality, and subagent dispatch isolation. (ref: ADR-027)
