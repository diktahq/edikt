# Architecture

[edikt:directives:start]: #
- Coding standards, security rules, and architecture patterns MUST be installed as .claude/rules/ files where Claude automatically reads and enforces them. (ref: ADR-001)
- Plans MUST be the persistent execution state — progress tracked directly in the plan file, surviving context compaction. (ref: ADR-001)
- edikt MUST NOT have runtime dependencies — installation is copy .md files only. (ref: ADR-001)
- edikt MUST NOT make silent background writes — suggestions fire only on strong signals; preferences and implementation details are ignored. (ref: ADR-001)
- edikt MUST NOT perform execution orchestration — Claude Code handles worktrees, agents, and parallelism natively. (ref: ADR-001)
- edikt MUST NOT use proprietary formats — everything MUST be plain markdown and YAML. (ref: ADR-001)
- A rule override file at `.edikt/rules/{name}.md` SHOULD NOT have the `<!-- edikt:generated -->` marker (so upgrade skips it). (ref: ADR-005)
- Both config-based and file-based agent customization options MUST be supported. (ref: ADR-005)
- Both the base pack and the extension MUST be version-controlled. (ref: ADR-005)
- Config MUST take precedence over file-based markers for agent customization. (ref: ADR-005)
- Custom agents (not from edikt templates) MUST always be skipped by upgrade — they have no matching template to compare against. (ref: ADR-005)
- If a rule override file exists at `.edikt/rules/{name}.md`, edikt MAY use it instead of the built-in pack. (ref: ADR-005)
- The extension file MUST be concatenated after the built-in pack when rules are installed. (ref: ADR-005)
- The project MUST be able to override any template by placing a file in `.edikt/templates/`. (ref: ADR-005)
- `/edikt:doctor` MUST report which rules are overridden, extended, or default. (ref: ADR-005)
- `/edikt:doctor` MUST report which templates are custom vs default. (ref: ADR-005)
- A tier-1 markdown command that invokes `bin/edikt <subcommand>` MUST detect absence of the binary on PATH before invocation and emit a clear actionable error directing the user to `edikt install <helper>` (or to the documented install path for the helper in question). (ref: ADR-029)
- Absent the `on_absent:` frontmatter field, the default MUST be `refuse-and-direct-user` (fail-closed). (ref: ADR-029)
- Adding a new orchestration verb to this list MUST require an ADR amending ADR-029 — the list is exhaustive, not illustrative. (ref: ADR-029)
- Each tier-1 command that invokes a tier-2 binary MUST document its failure-mode in command frontmatter. (ref: ADR-029)
- Helper install (including the `edikt` binary itself) MUST be a separate explicit step from tier-1 install. (ref: ADR-029)
- Other binary calls MUST remain forbidden in tier-1. (ref: ADR-029)
- Output may be displayed verbatim to the user (passthrough is fine), but parsing — extracting fields, splitting on whitespace, regex-matching against the binary's output — MUST NOT be done. (ref: ADR-029)
- The absence-detection check MUST cite the helper by name in the error so the user knows what to install. (ref: ADR-029)
- The markdown command MUST NOT depend on the binary's output SHAPE. (ref: ADR-029)
- Tier-1 MUST install without tier-2. (ref: ADR-029)
- Tier-1 markdown commands MAY invoke tier-2 binaries via `bin/edikt <subcommand>` for orchestration purposes, provided the four rules below all hold. (ref: ADR-029)
- `install.sh` MUST NOT install the Go binary; users MUST get tier-1 markdown and nothing else by default. (ref: ADR-029)
- ADR-029 Rule 3 MUST include `bin/edikt sidecar <subcommand>` as a permitted orchestration verb group. (ref: ADR-031)
- All four ADR-029 rules MUST hold for every `sidecar` subcommand invocation. (ref: ADR-031)
- Exit code only — Step 8 passes output verbatim and MUST NOT parse its shape. (ref: ADR-031)
- `on_absent: refuse-and-direct-user` MUST be declared in `/edikt:adr:enrich`'s frontmatter. (ref: ADR-031)
- ADR-029 Rule 3 MUST broaden the entry `bin/edikt gov compile` into the group permit `bin/edikt gov <subcommand>`. (ref: ADR-033)
- All four ADR-029 rules MUST hold for every `gov` subcommand invocation. (ref: ADR-033)
- Exit code MUST be the only contract — Phase 11.5 callers pass output verbatim and MUST NOT parse its shape. (ref: ADR-033)
- Tier-1 markdown commands invoking `bin/edikt gov <subcommand>` MUST declare `tier_2_dependency: edikt` in frontmatter. (ref: ADR-033)

## Prohibitions
- MUST NOT allow tier-1 to call tier-2 freely with no contract — tier boundary becomes meaningless and tier-2 helpers must keep output formats stable forever to avoid breaking tier-1 callers. (ref: ADR-029)
- MUST NOT fold migrated logic into existing enumerated verbs (e.g. fold compile-history into `gov compile`) — conflates pure deterministic state-machine work with the broader compile pipeline and loses the tight test surface. (ref: ADR-033)
- MUST NOT implement orchestration verbs (such as verify) as tier-1 markdown commands — markdown cannot reliably execute shell commands with timeouts, capture per-criterion exit codes, and persist a structured report. (ref: ADR-029)
- MUST NOT leave Python heredocs in tier-1 markdown commands — heredocs bypass every tier-2 invariant (not unit-tested, not benchmarked, not in the ADR-030 LLM-agnostic gate). (ref: ADR-033)
- MUST NOT pin each new `gov` subcommand individually via successive ADRs — the group-permit precedent set by ADR-031 covers the whole `gov` parent. (ref: ADR-033)
- MUST NOT route tier-1 to tier-2 calls through a Bash wrapper — the wrapper is itself a runtime dependency and the problem moves rather than disappearing. (ref: ADR-029)
- MUST NOT use a unified override model as the single extensibility mechanism — rules need extension (add to defaults), not just override; agents need per-file control. (ref: ADR-005)
