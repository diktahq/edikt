# Hooks

- Use PostToolUse hooks for auto-formatting after Write or Edit. Fire only on known source file extensions. Respect `EDIKT_FORMAT_SKIP=1` and config disable. (ref: ADR-003)
- Use shell preprocessing in slash commands to inject live project state (next ADR/INV/PRD number, active plan phase). NEVER guess or hardcode sequential identifiers. (ref: ADR-003)
- Agents run in forked subagents (`context: fork`), invoked by edikt on domain signals. They do not coordinate parallel execution — that is Claude Code's responsibility. (ref: ADR-003)
- CLAUDE.md sentinels use visible markdown link reference definitions: `[edikt:start]: #` and `[edikt:end]: #`. NEVER use HTML comment sentinels. (ref: ADR-006)
- Detect both formats (HTML comments and link references) for backward compatibility during init and upgrade. (ref: ADR-006)
- Hook JSON-wrapping migration MUST land in v0.5.0. All plaintext-emitting edikt hooks MUST emit JSON conforming to the Claude Code hook protocol by the v0.5.0 release. (ref: ADR-014)
- Hook migration MUST preserve user-visible message content byte-for-byte inside the JSON wrapping. NEVER reword user-facing strings as part of the wrapping migration. Content changes are separate commits with separate fixtures. (ref: ADR-014)
- `fixtures.yaml` §9.1 MUST document the resolution of the 3 ADR-011-removed fixture pairs: `pre-compact` permanently removed (hook deleted); `session-start-with-edikt` and `subagent-stop-critical` deferred with explicit blocker notes. Full re-addition is a v0.5.x follow-up, not a v0.5.0 ship-gate. (ref: ADR-014)
- Fixture expected outputs for migrated hooks MUST be regenerated by running the updated hook against its payload. NEVER hand-author expected outputs. (ref: ADR-014)
- New hook behavior refinements (`updatedInput` transforms, plan-phase tracking, new events `SessionEnd`/`SubagentStart`/`TaskCompleted`/`WorktreeCreate`/`WorktreeRemove`) are permissible in v0.5.0 provided each is covered by a characterization fixture pair before ship. (ref: ADR-014)
- ADR-011's v0.6.0 gate on hook protocol migration is LIFTED by this ADR. The characterization test gate opt-out polarity, `_runner.sh` immutability, and `_staged_runner.sh` extension rules from ADR-011 carry forward unchanged. (ref: ADR-014)
- `templates/settings.json.tmpl` MUST include an explicit `permissions` block with populated `deny`, `allow`, and `defaultMode: askBeforeAllow` keys. (ref: ADR-017)
- The `permissions.deny` list MUST include at minimum the destructive Bash patterns, destructive git patterns, plaintext HTTP fetch patterns, and sensitive file read patterns enumerated in ADR-017. (ref: ADR-017)
- The `permissions.allow` list MUST include edikt's operational tools (git, gh, pytest, npm test, `./test/run.sh`, `./test/test-e2e.sh`, make test, uv, ruff, WebFetch(https://**), WebSearch). (ref: ADR-017)
- Managed-region integrity for `settings.json` MUST use the sidecar mechanism at `~/.edikt/state/settings-managed.json`. NEVER embed `_edikt` metadata inside the settings.json itself. (ref: ADR-017, INV-005)
- On upgrade with a drifted `settings.json`, the writer MUST prompt the user before overwriting managed keys. User-added top-level keys outside the managed region MUST be preserved untouched. (ref: ADR-017)
- Security-driven rewrites MAY change user-visible hook content AND JSON wrapping in the same commit, but ONLY for: `templates/hooks/subagent-stop.sh`, `templates/hooks/file-changed.sh`, `templates/hooks/headless-ask.sh`, `templates/hooks/stop-failure.sh`. (ref: ADR-019)
- The carve-out applies only when the existing hook content violates INV-003 or INV-004. Style rewrites, UX polish, or unrelated changes MUST follow ADR-014 unaltered. (ref: ADR-019)
- Fresh characterization fixtures MUST be regenerated by running the updated hook against its payload, per ADR-011. Hand-authored expected outputs remain forbidden. (ref: ADR-019)
- Any future hook content change outside the four listed hooks MUST follow ADR-014's byte-for-byte rule. This carve-out does NOT generalize. (ref: ADR-019)
