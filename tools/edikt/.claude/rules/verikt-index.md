### Step 2 — Language

Ask: "Go or TypeScript?"

- **go** — Go module. Also ask for the module path (e.g. github.com/myorg/my-service). Collect as `--language go --module <value>`.
- **typescript** — TypeScript/Node.js. Also ask for the HTTP framework: Express (default), Fastify, or Hono. Collect as `--language typescript`. If not Express, add `--set HttpFramework=<value>`.

### Step 3 — Architecture

Present the options available for the chosen language:

**Go architectures:**
- **hexagonal** — ports & adapters; business logic isolated from infrastructure. Best for complex domains, long-lived services.
- **layered** — handler → service → repository → model. Familiar, predictable, lower ceremony.
- **clean** — concentric rings (entity → usecase → interface → infrastructure). Good for large teams with strict layering.
- **flat** — single package, no layer rules. Ideal for simple tools, CLIs, or prototypes.

**TypeScript architectures:**
- **hexagonal** — domain/ → application/ → infrastructure/ → transport/. Same boundaries as Go, TypeScript conventions.
- **flat** — single src/ directory. Good for simple services or prototypes.

Collect as `--arch <value>`.

### Step 4 — Capabilities
