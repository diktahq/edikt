**Key design decisions**

**Concurrency (scheduler.go)**
A buffered channel `sem` of capacity 3 acts as a counting semaphore. Before polling, `cap(s.sem) - len(s.sem)` determines free slots so we only claim as many jobs from the DB as we can immediately run — no goroutine pile-up between polls.

**Atomic job claiming (db.go:54)**
`UPDATE … WHERE id IN (SELECT … FOR UPDATE SKIP LOCKED)` is a single atomic operation. Multiple scheduler instances or rapid polls can never double-claim the same job — Postgres skips rows already locked by another transaction.

**HTTP routing (main.go)**
Uses Go 1.22's built-in method+path pattern matching (`GET /jobs/{id}`), so `r.PathValue("id")` works with no external router.

**Job type registration**
Call `sched.Register("your-type", fn)` before `sched.Run`. Unknown types are immediately marked `failed`.
