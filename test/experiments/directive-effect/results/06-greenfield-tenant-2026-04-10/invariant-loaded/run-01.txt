## Key design decisions

**Tenant isolation (per governance directives)**
- `internal/tenant` owns the single context key — tenant ID is only read from `ctx.Value`, never from request body/path/query string.
- `middleware.Tenant` extracts it from `X-Tenant-ID` (the auth boundary — replace with JWT validation for production).
- Every repository method takes `tenantID string` explicitly; `tenant_id` is in every `WHERE`, `INSERT`, and `JOIN`.
- Every `slog` call includes `"tenant_id", tid`.

**Polling invariant**
- `TenantsWithPending` first discovers which tenants have work, then `ClaimPending` runs a scoped query per tenant — no unscoped pass over all jobs at once.
- Workers call `tenant.NewContext` with the job's own `TenantID` before doing any work — they never run in a "no tenant" state.

**Concurrency**
- A buffered channel (`sem`, cap=3) acts as a semaphore. Before each poll tick, available slots are computed and only that many jobs are claimed, preventing over-claiming.
- Polls are non-reentrant (next tick waits for `poll()` to return) so the available-slot check is race-free.
- `SELECT ... FOR UPDATE SKIP LOCKED` in `ClaimPending` makes concurrent poller instances safe.

**Running it**
```bash
createdb jobscheduler
psql jobscheduler < migrations/001_create_jobs.sql
DATABASE_URL=postgres://localhost:5432/jobscheduler?sslmode=disable go run .

# Create a job
curl -X POST :8080/jobs \
  -H "X-Tenant-ID: acme" \
  -H "Content-Type: application/json" \
  -d '{"type":"send_email","payload":{"to":"a@b.com"}}'

# Check status
curl :8080/jobs/<id> -H "X-Tenant-ID: acme"
```
