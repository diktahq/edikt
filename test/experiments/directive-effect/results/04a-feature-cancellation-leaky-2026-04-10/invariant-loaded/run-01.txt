**`DELETE /orders/{id}`** — cancel an order within 1 hour of placement.

**Four files changed:**

**`internal/http/handlers/orders_cancel.go`** (new) — the handler, which:
1. Loads the order via `GetByID` (tenant-scoped automatically)
2. Verifies the caller owns the order — returns 404 to avoid leaking that an order exists for another user
3. Rejects if `CreatedAt` is older than 1 hour → 422
4. Rejects if status is not `paid` → 422 (catches already-cancelled, pending, fulfilled)
5. Calls `stripe.Refund` with idempotency key `order-{id}-refund`, mirroring the charge key pattern
6. Calls `MarkRefunded` to transition the order status
7. Enqueues a cancellation email (non-fatal, same pattern as order confirmation)
8. Returns 200 with the updated order

**`internal/repository/orders.go`** — added `MarkRefunded`, same tenant-scoped `UPDATE` pattern as `MarkPaid`.

**`internal/email/sender.go`** — added `SendCancellationConfirmation` and `OrderCancellationJob` (implements `queue.Job`).

**`internal/http/router.go`** — registered `DELETE /orders/{id}` behind the auth+logging middleware chain.

**Key design decisions aligned with the existing codebase:**
- Tenant isolation is enforced by the repository layer via context — the handler never touches `tenant_id` directly (INV-012)
- Stripe refund is synchronous on the request path; email is async via queue
- Idempotency key on the refund means retried cancellation requests don't double-refund
- The 404 for wrong-user (rather than 403) avoids leaking order ID existence across users within a tenant
