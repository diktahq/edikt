**`POST /orders/{id}/cancel`**

**`internal/repository/orders.go` — `MarkCancelled`**
Adds a conditional `UPDATE orders SET status = 'cancelled' WHERE id = $3 AND tenant_id = $4 AND status = 'paid'`. The `status = 'paid'` guard in the WHERE clause prevents double-cancellation in concurrent requests without needing a transaction.

**`internal/email/sender.go` — `SendOrderCancelled` + `OrderCancelledJob`**
Follows the same async queue pattern as order confirmation. Synchronous email on the request path is forbidden per the existing architecture.

**`internal/http/handlers/orders_cancel.go` — `CancelOrder`**
Validation sequence:
1. Fetch order by ID (tenant-scoped automatically by the repo)
2. Verify `order.UserID == sessionUserID` from context — returns 404 on mismatch to avoid leaking existence
3. Reject if status is not `paid` → 422
4. Reject if `time.Since(order.CreatedAt) > 1h` → 422
5. Call `stripe.Refund` with idempotency key `order-{id}-refund`
6. Call `MarkCancelled` — handles the race condition where two concurrent requests both pass step 3/4
7. Enqueue cancellation email (non-fatal if it fails)

**`internal/http/router.go`** — registers the route in the authenticated mux.
