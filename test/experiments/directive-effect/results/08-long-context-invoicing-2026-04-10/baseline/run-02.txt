## Invoicing feature — complete

Six files changed across the full stack. Here's the breakdown:

### New files

**`internal/domain/invoice.go`**
Two types: `Invoice` (ID, OrderID, UserID, TenantID, line items, subtotal/tax/total in cents, CreatedAt) and `InvoiceLineItem` (ProductID, Quantity, UnitCents, TotalCents). Monetary values follow the existing all-cents-as-`int64` convention.

**`internal/repository/invoices.go`**  
Three methods on `InvoicesRepo`:
- `Create` — opens a transaction, inserts the invoice row, then inserts each line item with `ExecContext`. `defer tx.Rollback()` ensures cleanup on any failure; only `tx.Commit()` makes it permanent. The invoice's `ID` is populated from `RETURNING id`, matching the orders repo pattern.
- `GetByID` — fetches the invoice row, then fetches its line items in a second query. Tenant-scoped like every other query in the codebase.
- `ListByUser` — returns invoices in reverse-chronological order, without line items (summary only). Initialised as `[]domain.Invoice{}` so it serialises as `[]` not `null` when empty.

The schema DDL is embedded as a Go comment at the top of the file since there's no migration system in the repo.

**`internal/service/invoice.go`**  
`InvoiceService` with three methods:
- `CreateFromOrder(ctx, order, cart)` — builds line items from cart items, calculates `subtotal`, applies 20% tax (`subtotal * 20 / 100`, integer truncation to avoid float rounding), derives `total = subtotal + tax`, then delegates to the repo. Logs on both success and failure via `slog`.
- `GetByID` — thin delegation to the repo.
- `ListByUser` — extracts `UserID` from context (same pattern as `CheckoutService`) and delegates.

**`internal/handler/invoice.go`**  
`InvoiceHandler` with two methods matching the existing handler style:
- `GET /invoices` → `List` — encodes the slice directly.
- `GET /invoices/{id}` → `Get` — uses `r.PathValue("id")` (Go 1.22, consistent with the `POST /checkout` route). Adds one deliberate improvement over the existing handlers: `sql.ErrNoRows` maps to `404` instead of `500`, since "invoice not found" is not a server error.

### Modified files

**`internal/service/checkout.go`**  
`CheckoutService` now takes `*InvoiceService` as a third constructor argument. After `orders.Create` succeeds, it calls `invoices.CreateFromOrder(ctx, order, cart)`. If invoice creation fails the checkout returns an error — the order exists in the DB at that point but the user gets a 500 and can retry.

**`cmd/main.go`**  
Wires the new layer: `NewInvoicesRepo` → `NewInvoiceService` → `NewInvoiceHandler`, passes `invoiceSvc` into `NewCheckoutService`, and registers the two new routes on the authenticated mux.
